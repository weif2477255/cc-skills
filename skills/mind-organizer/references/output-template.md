# 输出文档模板

本文档提供思维整理与任务规划的标准输出模板。

### 1.3 团队能力建设

## 模板使用说明

### 第一部分：思维整理总览

**目的**：将用户的口语化、碎片化想法转化为专业表达，并按四维度分类。

**要求**：
- 每个想法用一句话概括（20-30字）
- 使用专业术语，避免口语化表达
- 按四个维度分类：客户反馈和项目交付、产品技术演进、团队能力建设、战略规划
- 如果某个维度没有想法，可以省略该小节

**示例**：
- ❌ 口语化："我觉得我们应该搞一个自动化测试的东西"
- ✅ 专业化："建立自动化测试体系，提升回归测试效率"

### 第二部分：可行性分析

**目的**：基于思维锚点（工作职责、产品定位、团队状态）和四维度评估标准，对每个想法进行深度分析。

**分析结论标准**：
- ✅ **可行**：符合当前资源和战略，建议执行
- ⚠️ **需调整**：方向正确但需要调整范围、时机或方式
- ❌ **不建议**：不符合当前优先级或资源约束，建议暂缓

**核心理由要求**：
- 2-3句话说明结论的依据
- 必须基于 context.md 中的思维锚点
- 必须提及四维度中的至少2个维度
- 要具体，不要泛泛而谈

**优先级建议标准**：
- **P0**：必须立即执行（影响当前交付、严重线上问题、战略核心）
- **P1**：重要但可适当延后（影响下一阶段、战略相关、ROI高）
- **P2**：可以暂缓（无紧急压力、战略关联度低、ROI一般）

**关键风险**：
- 只列出真正需要关注的风险
- 如果没有明显风险，可以省略此项
- 风险要具体，并提示缓解方向

**示例**：

```markdown
### 想法：建立自动化测试体系

**分析结论**：⚠️ 需调整

**核心理由**：自动化测试对提升研发效能有显著价值（ROI高），但当前资源紧张，建议先聚焦核心模块的冒烟测试，而非全面覆盖。可以分阶段推进，先解决回归测试痛点。

**优先级建议**：P1

**关键风险**：初期投入较大，需要专人负责，建议从测试团队中指定1人专职推进。
```

### 第三部分：任务清单

**目的**：将高优先级想法转化为粗粒度的可执行任务。

**任务粒度标准**：
- ✅ 粗粒度："开发自动化测试框架"
- ❌ 过细："编写测试用例1、编写测试用例2..."
- ❌ 过粗："提升研发效能"

**任务详情要求**：
- 说明具体要做什么（1-2句话）
- 不需要详细的实施步骤
- 可以提及关键技术或方法

**目标要求**：
- 说明要达成的结果
- 要具体，不要模糊

**验收目标要求**：
- 必须可衡量
- 可以是数字指标、功能完成度、文档产出等

**P0/P1/P2 分级原则**：
- **P0**：立即执行，通常来自"可行性分析"中优先级为 P0 的想法
- **P1**：次优先，通常来自"可行性分析"中优先级为 P1 的想法
- **P2**：待定，可以只列出任务名称，不需要详细信息

**示例**：

```markdown
#### 任务1：建立核心模块自动化冒烟测试

- **任务详情**：针对 MOM 核心模块（工单、排程、执行）开发自动化冒烟测试框架，使用 Pytest + Selenium 实现
- **目标**：每次发版前自动执行冒烟测试，快速发现核心功能问题
- **验收目标**：覆盖 20+ 核心业务场景，执行时间 < 10分钟，集成到 CI/CD 流程
```

## 注意事项

1. **做减法而非加法**：在资源紧张的情况下，要敢于说"不建议"，帮助用户聚焦
2. **基于锚点分析**：所有分析必须基于 context.md 中的思维锚点，不能脱离实际情况
3. **简洁实用**：不要罗列思维过程，直接给结论和理由
4. **具体明确**：避免模糊表达，要给出可操作的建议
5. **风险提示**：主动指出关键风险，但不要过度渲染

## 输出文件命名

建议使用以下命名格式：

```
思维整理-YYYY-MM-DD.md
```

例如：`思维整理-2026-02-01.md`
