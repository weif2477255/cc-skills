# 输出质量规范

本文档定义团队月度评价的输出要求、质量原则和检查清单。

## 一、输出要求

### 1.1 报告结构

月度评价报告必须包含以下五个部分：

```markdown
# [yyyy-mm] MOM事业部产品开发团队月度评价

## 一、团队整体评分
## 二、团队成员评分
## 三、优秀成员事迹（S、A+、A评级）
## 四、其他成员综合评价
## 五、团队改进建议
```

### 1.2 团队整体评分

**必需内容**：
- 四维度评分表格：工作量(20%)、工作深度(30%)、工作成果(30%)、协作贡献(20%)
- 每个维度：得分 + 简洁评价（1-2句话）
- 总分和评级

**格式要求**：
- 使用表格格式
- 包含维度、权重、得分、评价四列

### 1.3 团队成员评分

**必需内容**：
- 所有成员的四维度评分和总分
- 建议评级

**格式要求**：
- 表格格式，包含：序号、姓名、职责、工作量(20)、深度(30)、成果(30)、协作(20)、总分、建议评级
- **强制要求**：按总分从高到低排序
- 只输出表格，不输出评分依据和推理过程

### 1.4 优秀成员事迹

**适用对象**：S、A+、A评级成员

**必需内容**：
- 每个成员使用三级标题：`### 姓名（评级，总分）`
- **突出贡献**：3-4条，每条包含具体事实和数据
- **综合评价**：2-3句话，根据成员的团队岗位撰写

**格式要求**：
- 突出贡献使用无序列表，每条以加粗的贡献点名称开头
- 综合评价单独成段
- 避免空洞赞美，必须有具体事实支撑

### 1.5 其他成员综合评价

**适用对象**：非S/A+/A评级成员

**必需内容**：
- 表格格式，包含序号、姓名、综合评价三列
- 每个成员的评价控制在2-3句话

**评价结构**：
1. 肯定优点：基于工作日志数据，具体说明做得好的地方
2. 指出不足：客观指出存在的问题
3. 改进建议：给出中肯可行的改进建议

### 1.6 团队改进建议

**必需内容**：
- 团队层面改进建议：2-3条
- 排名末尾3名成员的改进重点：原因分析 + 改进建议

**格式要求**：
- 使用三级标题分隔"团队层面"和"个别成员改进重点"
- 团队建议使用有序列表
- 个别成员使用无序列表，格式：`**姓名**：原因分析。建议：改进措施`

## 二、质量原则

### 2.1 数据准确性

**原则**：所有评分和评价必须基于工作日志数据

**要求**：
- JSON数据是评分参考，不是最终答案
- 必须阅读实际工作内容进行综合判断
- 评分逻辑：读取JSON统计数据 → 阅读工作内容 → 综合判断给出具体分数
- 使用具体的数字和事实支撑观点
- 避免主观臆断和情绪化表达

### 2.2 评分公平性

**原则**：严格遵守评分标准和评级分布规则

**要求**：
- 四维度评分严格按照 [scoring-guide.md](scoring-guide.md) 标准
- 评级分布符合规则：
  - S级≤10%，且必须同时满足"总分≥95分"+"排名在前10%"+"有重大突破"
  - A+级≤10%，A级≤10%
  - B+、B、C级按得分区间分配
- 成员评分表格必须按总分从高到低排序

### 2.3 格式规范性

**原则**：输出格式必须符合要求，便于阅读和审核

**要求**：
- 严格遵守报告结构（五个部分）
- 表格格式正确，列对齐
- 标题层级正确（一级标题用于报告标题，二级标题用于五个部分，三级标题用于子部分）
- 所有 `<!-- -->` 注释必须清除
- 单次Edit内容≤3000字

### 2.4 语言专业性

**原则**：语言风格正式、客观、专业，面向改进

**要求**：
- 使用正式的书面语言
- 避免口语化表达
- 问题识别客观，改进建议具体可行
- 语言建设性，避免归咎个人
- 避免重复和冗余的描述

### 2.5 保密性

**原则**：保护敏感信息

**要求**：
- 不包含敏感的客户信息
- 不包含未公开的战略规划
- 工作日志文件编码必须为UTF-8

## 三、质量检查清单

完成报告生成后，必须执行以下检查：

### 3.1 结构完整性检查

- [ ] 报告包含五个部分：团队整体评分、团队成员评分、优秀成员事迹、其他成员综合评价、团队改进建议
- [ ] 每个部分的标题层级正确（二级标题）
- [ ] 所有 `<!-- -->` 注释已清除

### 3.2 数据准确性检查

- [ ] 团队整体评分的四维度得分与成员平均分一致
- [ ] 成员评分表格的总分 = 四维度之和
- [ ] 成员评分表格按总分从高到低排序
- [ ] 优秀成员事迹中的评级和总分与评分表格一致
- [ ] 其他成员综合评价包含所有非S/A+/A评级成员

### 3.3 评分公平性检查

- [ ] S级成员≤10%，且总分≥95分，排名在前10%，有重大突破
- [ ] A+级成员≤10%，且总分≥90分
- [ ] A级成员≤10%，且总分≥85分
- [ ] 评级分布符合规则
- [ ] 评分依据基于工作日志数据，不是主观臆断

### 3.4 格式规范性检查

- [ ] 团队整体评分使用表格格式
- [ ] 成员评分表格包含所有必需列
- [ ] 优秀成员事迹使用"突出贡献"+"综合评价"结构
- [ ] 其他成员综合评价使用表格格式
- [ ] 团队改进建议使用三级标题分隔

### 3.5 内容质量检查

- [ ] 团队整体评价每个维度有1-2句话的评价说明
- [ ] 优秀成员事迹每人有3-4条突出贡献，包含具体事实和数据
- [ ] 优秀成员事迹的综合评价根据成员岗位撰写（参考 [team-structure.md](team-structure.md)）
- [ ] 其他成员综合评价遵循"肯定优点 + 指出不足 + 改进建议"结构
- [ ] 团队改进建议包含团队层面2-3条和排名末尾3名成员的改进重点
- [ ] 所有评价避免空洞赞美，有具体事实支撑

### 3.6 语言专业性检查

- [ ] 语言风格正式、客观、专业
- [ ] 避免口语化表达
- [ ] 改进建议具体可行，语言建设性
- [ ] 无重复和冗余的描述
