# 团队绩效评分指南

本文档说明团队月报中的绩效评分标准和规则。**重要**：JSON数据是评分参考，AI必须阅读实际工作内容进行综合判断。

## JSON参考数据说明

清洗脚本生成的JSON文件包含`scoring_reference`字段，提供评分参考：

**核心字段**：
- `weighted_total`：关键词加权总数，工作量评分的核心参考
- `coverage`：工作分布（实际天数/总天数），反映工作连续性
- `workload_score`：工作量维度的**参考分数**（非最终分数）
- `depth_high_count/depth_medium_count`：高/中难度关键词次数，工作深度参考
- `result_high_count/result_medium_count`：高/中价值成果关键词次数，工作成果参考
- `collab_high_count/collab_medium_count`：高/中贡献协作关键词次数，协作贡献参考

**使用原则**：
- JSON数据提供定量参考，不是最终答案
- 必须阅读`details`字段的实际工作内容
- 综合判断：定量指标（JSON）+ 定性分析（内容）→ 给出具体分数和依据

## 四维度评分体系

| 维度 | 权重 | 满分 | 核心评估点 |
|------|------|------|-----------|
| 工作量 | 20% | 20 | 关键词加权总数、工作分布、涉及范围 |
| 工作深度 | 30% | 30 | 技术难度、问题复杂度、创新性 |
| 工作成果 | 30% | 30 | 功能交付、版本发布、客户价值 |
| 协作贡献 | 20% | 20 | 跨团队协作、知识分享、团队支持 |

**总分** = 工作量 + 工作深度 + 工作成果 + 协作贡献（满分100分）

## 评级标准

采用**6档评级体系**，结合**得分区间**和**比例分配**双重标准：

| 得分范围 | 评级 | 定义 | 比例要求 |
|----------|------|------|----------|
| X≥95 | S | 卓越 | ≤10% |
| 90≤X<95 | A+ | 优秀+ | ≤10% |
| 85≤X<90 | A | 优秀 | ≤10% |
| 80≤X<85 | B+ | 良好+ | 按得分分配 |
| 75≤X<80 | B | 良好 | 按得分分配 |
| X<75 | C | 待改进 | 按得分分配 |

### S级判定逻辑（必须同时满足）

1. **总分≥95分**
2. **排名在前10%**
3. **满足以下至少一条重大突破**（需在评语中引用具体证据）：

**技术攻关**：
- 解决了阻塞团队/产品的关键技术难题（如性能瓶颈、架构缺陷）
- 攻克了行业内公认的技术难点（如复杂算法、分布式一致性问题）
- 技术方案被团队采纳为标准实践

**重大功能**：
- 交付了影响产品核心竞争力的关键功能（如KMMOM核心模块、低代码平台关键能力）
- 完成了客户合同中的里程碑级交付物
- 功能上线后带来可量化的业务价值（如客户数量、使用率显著提升）

**创新方案**：
- 提出并实施了显著提升效率的创新方案（如开发效率提升30%+、部署时间减少50%+）
- 设计了可复用的技术框架或工具，被多个项目采用
- 创新方案获得管理层认可或在团队内推广

**重要**：缺少任何一个条件都不能评S级（即使95分但无重大突破 → A+）。

## 评级分配算法

1. **按总分排序**：将所有成员按总分从高到低排序
2. **分配S级**：前10%且得分≥95分 → S级
3. **分配A+级**：S级之后的10%且得分≥90分 → A+级
4. **分配A级**：A+级之后的10%且得分≥85分 → A级
5. **分配B+/B/C级**：剩余成员按得分区间分配

### 分布示例（18人团队）

- S级：前2名（11.1%），得分≥95分
- A+级：第3-4名（11.1%），得分≥90分
- A级：第5-6名（11.1%），得分≥85分
- B+级：第7-12名，得分80-89分（含得分≥85但超出A级配额的成员）
- B级：第13-17名，得分75-80分
- C级：第18名，得分<75分

### 特殊情况处理

1. **得分不满足评级要求**：如果前10%成员得分<95分，则不评S级，顺延至A+级
2. **同分处理**：如果边界位置出现同分，优先考虑工作深度和成果维度得分更高者
3. **比例微调**：如果严格10%导致不合理（如18人团队1.8人），可向上取整至2人（11.1%）

## 详细评分标准

### 1. 工作量（20分）

**评分逻辑**：
1. 读取JSON参考：`weighted_total`（加权总数）、`coverage`（工作分布）
2. 阅读工作内容：涉及模块数量、任务复杂度、工作范围
3. 综合判断：JSON数据提供基准，内容分析进行调整

**评分标准**（按21个工作日计）：
- **18-20分**：weighted_total≥80 + coverage≥75% + 涉及≥5个模块 + 内容显示工作饱满
- **15-17分**：weighted_total 60-79 + coverage 55-70% + 涉及3-4个模块 + 内容显示工作正常
- **12-14分**：weighted_total 40-59 + coverage 40-50% + 涉及1-2个模块 + 内容显示工作一般
- **<12分**：weighted_total<40 + coverage<40% + 范围狭窄 + 内容显示工作不足

**判断要点**：
- 如果weighted_total高但内容显示重复性工作 → 适当降分
- 如果weighted_total中等但内容显示高质量工作 → 适当加分
- 如果coverage低但有合理原因（如请假、出差）→ 说明情况，不过度扣分

**示例**：
- 张三：JSON显示weighted_total=85, coverage=86%（参考分19分）→ 阅读内容：涉及KMMOM核心模块、低代码平台、安装工具3条产品线，工作饱满 → 综合判断：**19分**（JSON参考准确，内容支持高分）
- 李四：JSON显示weighted_total=68, coverage=71%（参考分16分）→ 阅读内容：主要集中在审批流单一模块，但实现质量高 → 综合判断：**16分**（JSON参考合理，内容未发现需调整因素）

### 2. 工作深度（30分）

**评分逻辑**：
1. 读取JSON参考：`depth_high_count`（高难度关键词次数）、`depth_medium_count`（中等难度关键词次数）
2. 阅读工作内容：技术难度、问题复杂度、创新性
3. 综合判断：关键词统计提供线索，内容分析确定难度

**关键词识别标准**：

| 难度 | 权重 | 关键词示例 |
|------|------|-----------|
| 高难度 | ×2 | 架构、重构、设计模式、系统设计、优化、性能、调优、瓶颈、算法、分布式、高并发、并发、多线程、异步、安全、底层、内核、框架、中间件、引擎 |
| 中等难度 | ×1 | 模块、组件、插件、扩展、功能模块、接口、集成、API、对接、接口开发、接口设计、配置、部署、迁移、升级、安装、环境 |
| 低难度 | - | 修复、调整、更新、简单功能 |

**评分标准**：
- **27-30分**：多项高难度任务（depth_high≥8）+ 跨系统问题 + 有技术创新
- **24-26分**：部分高难度任务（depth_high 4-7）+ 跨模块问题 + 改进方案
- **21-23分**：主要为中等难度（depth_medium≥10）+ 模块内问题 + 常规实现
- **18-20分**：中低难度混合 + 简单跨模块问题 + 少量改进
- **<18分**：主要为低难度 + 单模块简单问题 + 无创新

**判断要点**：
- 关键词统计是线索，必须阅读内容确认真实难度
- 如果关键词多但内容显示简单配置 → 降分
- 如果关键词少但内容显示复杂问题解决 → 加分

**示例**：
- 王五：JSON显示depth_high=10, depth_medium=8 → 阅读内容：架构级改造2项（KMMOM架构优化、分布式缓存设计）+ 核心模块3个 → 综合判断：**27分**（高难度任务确实存在）
- 赵六：JSON显示depth_high=2, depth_medium=12 → 阅读内容：模块级功能3项 + 接口集成5个 → 综合判断：**23分**（中等难度为主，符合预期）

### 3. 工作成果（30分）

**评分逻辑**：
1. 读取JSON参考：`result_high_count`（高价值成果关键词次数）、`result_medium_count`（中等价值成果关键词次数）
2. 阅读工作内容：功能交付、版本发布、客户价值
3. 综合判断：关键词统计提供线索，内容分析确定价值

**关键词识别标准**：

| 价值 | 权重 | 关键词示例 |
|------|------|-----------|
| 高价值 | ×2 | 完成、交付、发布、上线、投产、提交、里程碑、版本、正式环境、生产环境、发版、客户、用户、需求、验收、客户需求、用户需求 |
| 中等价值 | ×1 | 开发、实现、编码、编写、功能开发、测试通过、测试、验证、联调、集成、集成测试 |
| 低价值 | - | 调研、讨论、计划、准备 |

**评分标准**：
- **27-30分**：多个重要功能（result_high≥12）+ 关键里程碑 + 高客户价值
- **24-26分**：部分重要功能（result_high 8-11）+ 参与发布 + 中等价值
- **21-23分**：常规功能（result_high 4-7）+ 日常迭代 + 一般价值
- **18-20分**：少量功能（result_high 1-3）+ 部分完成 + 较低价值
- **<18分**：交付很少 + 无明显成果 + 低价值

**判断要点**：
- 关键词统计是线索，必须阅读内容确认真实价值
- 如果关键词多但内容显示未完成交付 → 降分
- 如果关键词少但内容显示高价值交付 → 加分

**示例**：
- 孙七：JSON显示result_high=15, result_medium=10 → 阅读内容：交付安装工具优化（客户反馈好评）+ 支撑2产品线上线 → 综合判断：**26分**（高价值成果确实存在）
- 周八：JSON显示result_high=6, result_medium=12 → 阅读内容：审批流模块开发 + 测试通过 + 待上线 → 综合判断：**23分**（常规功能，符合预期）

### 4. 协作贡献（20分）

**评分逻辑**：
1. 读取JSON参考：`collab_high_count`（高贡献协作关键词次数）、`collab_medium_count`（中等贡献协作关键词次数）
2. 阅读工作内容：跨团队协作、知识分享、团队支持
3. 综合判断：关键词统计提供线索，内容分析确定贡献

**关键词识别标准**：

| 贡献 | 权重 | 关键词示例 |
|------|------|-----------|
| 高贡献 | ×2 | 协作、合作、配合、协同、团队协作、讨论、分享、培训、讲解、教学、技术分享、支持、帮助、协助、指导、审查、Review、Code Review、代码审查 |
| 中等贡献 | ×1 | 沟通、对接、协调、确认、同步、会议、评审、汇报 |
| 低贡献 | - | 独立完成、个人工作 |

**评分标准**：
- **18-20分**：频繁跨团队协作（collab_high≥10）+ 有培训分享 + 积极支持团队
- **15-17分**：有跨团队协作（collab_high 6-9）+ 有文档贡献 + 良好支持
- **12-14分**：团队内协作（collab_high 3-5）+ 偶尔分享 + 一般支持
- **9-11分**：少量协作（collab_high 1-2）+ 很少分享 + 支持较少
- **<9分**：协作很少 + 无分享 + 支持不足

**判断要点**：
- 关键词统计是线索，必须阅读内容确认真实贡献
- 如果关键词多但内容显示形式化协作 → 降分
- 如果关键词少但内容显示高质量支持 → 加分

**示例**：
- 吴九：JSON显示collab_high=12, collab_medium=8 → 阅读内容：产品协作2次 + 跨团队支持3次 + 技术分享1次 → 综合判断：**18分**（高贡献协作确实存在）
- 郑十：JSON显示collab_high=4, collab_medium=10 → 阅读内容：独立工作为主 + 偶尔沟通 → 综合判断：**13分**（一般协作，符合预期）

## 评分流程

1. **读取JSON文件**：加载清洗后的日志和评分参考数据
2. **阅读工作内容**：查看每个成员的`details`字段（必需步骤）
3. **四维度评分**：参考JSON统计数据 + 具体内容判断 → 给出具体分数和评分依据
4. **总分计算**：总分 = 四维度之和（0-100分）
5. **评级判定**：根据总分对应评级，S级需额外检查重大突破
6. **分布检查**：确保S级≤10%，A+级≤10%，A级≤10%
