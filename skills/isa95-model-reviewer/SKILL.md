# ISA-95 Model Reviewer

## Overview

用严格、挑错优先的方式评审制造业业务模型与数据模型。先判断业务场景与 ISA-95 的适配性，再评审模型定义是否标准、模型之间关系是否合理、活动闭环是否完整、追溯链是否成立，并给出可落地的建模调整建议。

## Review Posture

始终保持以下评审姿态：

- 先判断场景，再判断模型；不要脱离业务意图机械对字段打勾。
- 严格指出缺陷；不要用“基本合理”“大体可用”掩盖关键问题。
- 明确区分 4 类结论：`标准不匹配`、`建模缺陷`、`工程风险`、`信息不足`。
- 每指出一个缺陷，都要说明“为什么错、会造成什么影响、应该如何改”。
- 如果业务场景并不适合强行套 ISA-95，直接说明不适合，停止机械映射七层模型。
- 如果用户要求结合互联网资料评判，或者该场景有明显的行业差异风险，主动先做外部调研再下结论。

## Inputs

优先读取用户给出的任何材料：

- 业务需求文档
- 数据模型设计文档
- ER 图、类图、领域模型图
- 表结构、DDL、字段定义
- 流程图、状态流转图、样例单据

如果信息不足以判断“关系是否合理”：

1. 先根据名称、流程、状态、上下游对象谨慎推断业务场景。
2. 再按 `references/scenario-research-guide.md` 做定向检索。
3. 仍不足时，明确列出假设与缺失信息，不要装作已经判断充分。

## Workflow

### Step 1: Judge ISA-95 applicability

先回答：这是不是运行管理类场景。

通常适合用 ISA-95 作为主框架的场景：

- 生产运行：生产计划、派工、执行、报工、产量、WIP、工单、工艺执行
- 质量运行：检验计划、检验执行、结果记录、不合格、处置、CAPA 闭环
- 维护运行：维护计划、工单、任务执行、故障反馈、备件消耗、设备状态
- 库存与物料运行：发料、退料、转移、批次、库存状态、物料追溯、在制品流转
- 运行分析：产能、效率、停机、质量损失、维护绩效、库存准确率

通常不应强行套用 ISA-95 七层的场景：

- 主数据管理
- 组织权限与账户体系
- 财务核算与总账
- 通用配置平台
- 纯报表门户或纯主数据字典

如果只是部分适用，也要明确写出“ISA-95 只适合作为局部参考”。

### Step 2: Build the scenario map

先建立场景图，再看模型：

- 业务目标是什么
- 谁触发流程，何时触发
- 核心业务对象是什么：订单、工单、任务、批次、资源、规范、样本、结果、事件
- 决策与执行发生在哪些层次：定义、计划、派工、执行、采集、跟踪、分析
- 上游依据是什么，下游结果是什么
- 需要什么追溯：来源、去向、责任、时间、版本、数量、批次、状态

如果用户没有明确说明业务场景，但模型名已经暴露意图，例如 `MaintenanceWorkOrder`、`InspectionResult`、`ProductionDispatchOrder`，就可以先推断，再在结论里写明推断依据。

### Step 3: Load the detailed rubric

阅读 `references/isa95-review-rubric.md`，按以下维度逐项判断：

- 适用性与边界是否清晰
- 模型定义是否标准
- 模型粒度是否一致
- 模型之间关系是否合理
- 活动闭环与追溯链是否成立
- 可扩展性与模式复用是否合理
- 业务场景特有约束是否被体现

### Step 4: Supplement with external evidence when needed

当出现以下任一情况时，优先联网调研再评审：

- 用户明确要求结合互联网资料给出总体判断
- 业务场景术语明显具有行业差异，例如离散制造、流程制造、批次质量、设备维护策略
- 你对某一对象或关系的行业常见做法不够确定
- 用户材料只给了局部模型，缺少完整业务流程

调研要求：

- 先查官方或标准来源，再查行业组织或主流厂商资料
- 明确区分：`标准依据`、`行业实践`、`基于场景的推断`
- 不要把营销口号当成建模标准

调研方法详见 `references/scenario-research-guide.md`。

### Step 5: Map the model set to the operating chain

优先按以下链条审视，而不是按文档章节顺序审视：

- 资源层：谁可被调度、执行、消耗、占用
- 定义层：标准、规范、工艺、检验规范、维护规程、能力定义
- 计划层：计划、请求、排程、需求分解、任务准备
- 执行层：派工、工单、执行任务、执行反馈、结果确认
- 数据收集层：事实记录、过程参数、样本数据、事件、消耗记录
- 跟踪层：进度、状态快照、在制状态、设备状态、库存状态
- 分析层：绩效、偏差、损失、趋势、统计汇总

不要机械要求七层全有。只有在业务确实需要时，缺失某层才构成问题。真正严重的是：

- 计划无法落到执行
- 执行无法回写事实
- 事实无法追溯来源
- 状态无法解释由哪些事件导致
- 分析直接篡改或替代事务事实

### Step 6: Review model definitions

重点抓这些硬伤：

- 把“定义”与“实例”混成一个模型
- 把“计划”与“执行结果”混成一个模型
- 把“状态快照”与“事件事实”混成一个模型
- 把“分析汇总”当成业务事实主表
- 用一个模型同时承担多个生命周期阶段
- 名称很像，但语义边界不同或相互重叠
- 缺少版本、有效期、组织归属、责任主体、时间语义、单位或批次语义

### Step 7: Review relationships aggressively

关系评审是本 skill 的重点。至少回答以下问题：

- 下游对象是否能追溯到上游依据
- 关系方向是否正确：一般是下游引用上游依据，而不是反向硬绑
- 一对多、多对多是否被明确建模，而不是偷塞在文本字段里
- 是否缺少 Link/明细/事件模型来承接真实关系
- 是否把“包含关系”“引用关系”“派生关系”“统计关系”混为一谈
- 是否存在跨领域直接耦合，导致模型边界失控
- 是否有“一个模型引用所有东西”的上帝对象
- 是否缺少版本关系，导致执行无法追溯到当时生效的定义
- 是否缺少来源链、去向链、责任链、批次链、状态变化链

对于关系问题，优先给出结构性建议，例如：

- 新增 Link 模型而不是继续堆外键
- 拆分“请求/计划/执行/结果”四个阶段对象
- 引入事件模型承接状态变化，而不是只保留当前状态
- 引入版本化定义对象，执行对象只引用生效版本
- 分离事务事实表与分析汇总表

### Step 8: Grade the findings

按以下级别输出：

- `严重缺陷`：会直接破坏业务边界、追溯能力、闭环完整性、数据一致性或后续扩展能力
- `重要缺陷`：当前可实现，但会显著增加歧义、耦合、返工成本或审计风险
- `建议优化`：不修也能跑，但会降低复用性、一致性、可读性或长期演进能力
- `待确认问题`：材料不足，必须补充后才能下结论

### Step 9: Generate the report

使用 `template.md` 组织输出。报告必须至少包含：

- 业务场景判断
- 是否适合使用 ISA-95 作为主框架
- 评审依据与来源链接
- 总体结论
- 严重缺陷
- 重要缺陷
- 建议优化
- 推荐的模型调整方案
- 假设与待补充信息

## Common High-Severity Defects

看到以下模式时，优先判定为严重缺陷：

- `工艺定义` 与 `工艺执行记录` 复用同一个模型
- `计划单` 直接兼任 `派工单` 或 `完工单`
- 只有当前状态，没有状态事件或关键时间点
- 执行记录无法追溯到计划、定义版本、资源、物料批次或责任人
- 分析报表模型直接承担事务写入
- 质量结果与质量判定、不合格处置、CAPA 责任链揉成一个大表
- 维护工单、维护任务、故障记录、备件消耗不分层
- 库存数量、库存批次、库存移动、库存状态快照混成一个模型
- 为了“少建几个模型”而牺牲可追溯性和边界清晰度

## Scenario Notes

### Production operations

重点看：工艺定义、工单/派工、执行记录、物料消耗、批次/序列、在制状态、产量反馈、性能分析是否形成闭环。

### Quality operations

重点看：规范/检验计划、检验执行、样本、结果、不合格、处置、复检、CAPA 是否被错误合并或断链。

### Maintenance operations

重点看：设备/资产、维护策略或规程、维护请求、维护工单、任务执行、故障事件、备件与工时反馈、健康或绩效分析是否分层清楚。

### Inventory and material operations

重点看：物料定义、库存单元、批次、库存移动、预留/分配、发退料、在制流转、状态快照、追溯链是否被区分清楚。

## Use Local Method Guidance When Available

如果当前工作区包含与 ISA-95 整体化设计方法相关的内部文档，优先把它当作团队方法论补充，再用本 skill 做严格评审。典型补充点包括：

- 整体优先，而不是逐条翻译需求
- 识别通用模式与差异点
- 先梳理活动链，再决定模型集合
- 把可复用结构沉淀成模板，把差异点隔离为业务扩展
