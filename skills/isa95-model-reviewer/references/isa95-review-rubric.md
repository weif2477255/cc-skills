# ISA-95 Model Review Rubric

## How to use this rubric

先判断场景，再按维度逐项评审。不要把所有问题都说成“规范问题”。把问题分成：场景不适配、概念错误、关系错误、追溯断裂、工程风险、信息不足。

## Dimension 1: Scenario fit

### Questions

- 这是运行管理类业务，还是主数据/配置/财务/权限类业务？
- 如果属于运行管理，它主要落在生产、质量、维护、库存中的哪一类？
- 是否存在明显的跨领域混合场景？如果有，边界是否划清？

### Severe defects

- 明显不属于运行管理类场景，却硬套 ISA-95 七层
- 明明是混合场景，却没有明确主线和边界
- 用一个核心模型覆盖多个领域对象，导致概念失真

## Dimension 2: Definition standardization

### Questions

- 模型名是否表达单一稳定语义？
- 是否清楚区分定义、计划、执行、事件、快照、分析？
- 是否具备必要的标识、状态、时间、组织、责任、版本、单位、批次语义？
- 同类模型是否采用相近结构，而不是各写一套？

### Severe defects

- 定义对象与执行对象混用
- 当前状态替代事件事实
- 单据头、明细、结果、统计塞进一个模型
- 缺失关键语义字段，导致对象不可识别或不可追溯

## Dimension 3: Relationship rationality

### Questions

- 下游对象是否能追溯到上游依据？
- 关系方向是否正确？
- 多对多关系是否显式建模？
- 是否区分包含、引用、派生、汇总、版本、生效关系？
- 是否存在“上帝对象”或异常跨域耦合？

### Severe defects

- 派工或执行对象找不到来源计划或定义版本
- 批次、样本、故障、消耗等关键关系用字符串拼接保存
- 一对多/多对多关系没有 Link 或明细承接
- 汇总关系与事务关系混写

## Dimension 4: Closed-loop completeness

### Questions

- 资源、定义、计划、执行四个必要环节是否成立？
- 需要事实采集时，是否有事件或记录层？
- 需要状态跟踪时，是否有快照或状态链？
- 需要分析时，是否明确分析来源是事务事实而不是人工维护？

### Severe defects

- 有计划无执行
- 有执行无事实记录
- 有状态无事件依据
- 有分析无来源数据

## Dimension 5: Traceability and auditability

### Questions

- 能否追溯来源、去向、责任人、时间点、版本、数量、单位、批次？
- 关键状态变化是否有事件链？
- 历史定义变更后，旧执行是否仍能解释？

### Severe defects

- 关键对象没有时间点或责任链
- 定义变更后历史执行失真
- 无法回答“为什么现在是这个状态”
- 无法回答“这条结果来自哪次执行、哪版定义、哪批物料”

## Dimension 6: Reuse and extensibility

### Questions

- 通用模式是否被抽出来复用？
- 差异点是否被隔离在扩展属性或扩展模型？
- 新增同类业务时，会不会只能复制粘贴再改名？
- 是否为了眼前需求破坏长期结构一致性？

### Severe defects

- 相似业务对象结构完全不同且没有合理解释
- 业务差异直接侵入主体模型骨架
- 新场景一来就必须推翻现有关系结构

## Layer checklist

### Resource layer

检查资源对象是否能表达：可用性、能力、归属、状态、时间有效性。

### Definition layer

检查定义对象是否能表达：版本、生效、废弃、适用范围、标准参数、审批或发布语义。

### Plan layer

检查计划对象是否能表达：需求来源、计划时间、计划数量、计划资源、优先级、偏差比较依据。

### Execution layer

检查执行对象是否能表达：来源依据、执行主体、开始结束、执行状态、执行结果、异常或中止原因。

### Collection layer

检查事实记录是否能表达：采集时间、采集来源、业务关联对象、数量/参数/结果、样本或批次信息。

### Tracking layer

检查快照对象是否能表达：状态时间点、状态来源、当前值与上一状态的关联、解释链。

### Analysis layer

检查分析对象是否明确是派生层，而不是事务源；并能说明口径、时间窗、来源事实。

## Domain-specific checkpoints

### Production

- 是否区分工艺定义、生产计划、派工、执行记录、产量记录、物料消耗、在制跟踪、绩效分析
- 是否能追溯到工艺版本、工单来源、物料批次、设备/工位、操作者
- 是否把“工单”当成万能总表

### Quality

- 是否区分规范、检验计划、检验任务、样本、结果、不合格、处置、CAPA
- 是否把“结果”与“判定/处置”混成一表
- 是否缺少复检、让步接收、隔离、责任闭环等关系

### Maintenance

- 是否区分设备/资产、维护策略/规程、维护请求、维护工单、任务、故障、备件、工时、执行反馈
- 是否把“故障事件”与“维护执行”混成一表
- 是否能支撑 MTBF、MTTR、停机损失等分析

### Inventory and material

- 是否区分物料定义、库存单元、批次、库存移动、预留/分配、发退料、库存快照
- 是否把当前库存量与移动流水混成同一事实模型
- 是否能支持来源去向、批次追溯、在制流转

## Issue grading guide

### 严重缺陷

会导致以下至少一项成立：

- 业务边界错误
- 上下游关系错误
- 无法追溯或无法审计
- 生命周期混乱
- 扩展到同类场景时高概率返工

### 重要缺陷

当前可以落地，但会导致以下至少一项成立：

- 概念歧义大
- 关系理解成本高
- 后续实现复杂度显著升高
- 测试、报表、集成、审计成本升高

### 建议优化

结构不够优雅或不利于复用，但短期内不阻断交付。

## Recommendation patterns

看到问题时，优先从以下模式中选择建议：

- 拆分定义对象与实例对象
- 拆分计划对象与执行对象
- 引入明细或 Link 模型承接多对多关系
- 引入事件模型承接状态变化
- 引入快照模型承接状态查询优化
- 引入版本化定义并让执行引用生效版本
- 分离事务事实层与分析汇总层
- 抽取通用模板字段，隔离业务差异字段
